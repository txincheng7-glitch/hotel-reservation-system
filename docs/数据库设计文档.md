# 酒店后端管理系统数据库设计文档

## 1. 数据库设计概述

### 1.1 设计目标
- 为酒店后端管理系统提供高效、稳定的数据存储方案
- 支持酒店、房型、预订等核心业务功能
- 确保数据完整性和一致性
- 优化查询性能，提高系统响应速度

### 1.2 设计原则
- **规范化**：遵循数据库规范化原则，减少数据冗余
- **完整性**：使用约束确保数据完整性
- **性能**：合理设计索引，优化查询性能
- **可扩展性**：支持系统功能的扩展和未来业务的增长

### 1.3 技术选型
- **数据库**：MySQL 8.0+
- **存储引擎**：InnoDB（支持事务和外键）
- **字符集**：utf8mb4（支持中文和特殊字符）
- **排序规则**：utf8mb4_unicode_ci（通用排序规则）

## 2. 数据库表结构

### 2.1 用户表（users）

| 字段名 | 数据类型 | 约束 | 描述 |
|--------|----------|------|------|
| `id` | `bigint UNSIGNED` | `NOT NULL AUTO_INCREMENT PRIMARY KEY` | 用户ID |
| `username` | `varchar(50)` | `NOT NULL UNIQUE` | 用户名 |
| `password` | `varchar(255)` | `NOT NULL` | 密码哈希值 |
| `email` | `varchar(100)` | `UNIQUE` | 邮箱 |
| `phone` | `varchar(20)` | `UNIQUE` | 手机号 |
| `role` | `enum('user','merchant','admin')` | `NOT NULL DEFAULT 'user'` | 用户角色 |
| `created_at` | `datetime` | `NOT NULL DEFAULT CURRENT_TIMESTAMP` | 创建时间 |
| `updated_at` | `datetime` | `NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 更新时间 |

**索引**：
- `PRIMARY KEY`：`id`
- `UNIQUE`：`username`、`email`、`phone`

### 2.2 酒店表（hotels）

| 字段名 | 数据类型 | 约束 | 描述 |
|--------|----------|------|------|
| `id` | `bigint UNSIGNED` | `NOT NULL AUTO_INCREMENT PRIMARY KEY` | 酒店ID |
| `merchant_id` | `bigint UNSIGNED` | `NOT NULL` | 商家ID（外键） |
| `name` | `varchar(100)` | `NOT NULL` | 酒店名称 |
| `address` | `varchar(255)` | `NOT NULL` | 酒店地址 |
| `description` | `text` | | 酒店描述 |
| `star` | `tinyint UNSIGNED` | | 酒店星级 |
| `rating` | `decimal(2,1)` | `DEFAULT 0.0` | 酒店评分 |
| `opening_date` | `date` | | 开业日期 |
| `status` | `enum('pending','approved','rejected','published','unpublished')` | `NOT NULL DEFAULT 'pending'` | 酒店状态 |
| `audit_comment` | `text` | | 审核备注 |
| `is_deleted` | `tinyint(1)` | `NOT NULL DEFAULT 0` | 是否删除（软删除） |
| `created_at` | `datetime` | `NOT NULL DEFAULT CURRENT_TIMESTAMP` | 创建时间 |
| `updated_at` | `datetime` | `NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 更新时间 |

**索引**：
- `PRIMARY KEY`：`id`
- `INDEX`：`merchant_id`、`status`

**外键**：
- `merchant_id` 引用 `users(id)`，`ON DELETE RESTRICT ON UPDATE RESTRICT`

### 2.5 价格日历表（price_calendar）

| 字段名 | 数据类型 | 约束 | 描述 |
|--------|----------|------|------|
| `id` | `bigint UNSIGNED` | `NOT NULL AUTO_INCREMENT PRIMARY KEY` | 价格日历ID |
| `hotel_id` | `bigint UNSIGNED` | `NOT NULL` | 酒店ID（外键） |
| `room_id` | `bigint UNSIGNED` | `NOT NULL` | 房型ID（外键） |
| `date` | `date` | `NOT NULL` | 日期 |
| `price` | `decimal(10,2)` | `NOT NULL` | 价格 |
| `available` | `smallint` | `NOT NULL` | 可用房间数 |
| `status` | `enum('available','closed','full')` | `DEFAULT 'available'` | 状态 |

**索引**：
- `PRIMARY KEY`：`id`
- `UNIQUE`：`room_id, date`（确保每个房型每天只有一条记录）
- `INDEX`：`hotel_id, date`

**外键**：
- `hotel_id` 引用 `hotels(id)`，`ON DELETE RESTRICT ON UPDATE RESTRICT`
- `room_id` 引用 `rooms(id)`，`ON DELETE RESTRICT ON UPDATE RESTRICT`

| 字段名 | 数据类型 | 约束 | 描述 |

| `guest_info` | `json` | `NOT NULL` | 客人信息（JSON格式） |
| `contact_name` | `varchar(50)` | `NOT NULL` | 联系人姓名 |

## 3. 索引设计

### 3.1 主键索引
- 所有表都使用自增的 `id` 作为主键，确保唯一性和高效的查询

### 3.2 唯一索引
- `users(username)`：确保用户名唯一
- `users(email)`：确保邮箱唯一
- `users(phone)`：确保手机号唯一
- `tags(name)`：确保标签名称唯一
- `facilities(name)`：确保设施名称唯一
- `hotel_tags(hotel_id, tag_id)`：确保每个酒店的标签唯一
- `hotel_facilities(hotel_id, facility_id)`：确保每个酒店的设施唯一
- `price_calendar(room_id, date)`：确保每个房型每天只有一条价格记录

### 3.3 普通索引
- `hotels(merchant_id)`：优化根据商家查询酒店的性能
- `hotels(status)`：优化根据状态查询酒店的性能
- `hotel_images(hotel_id)`：优化根据酒店查询图片的性能
- `rooms(hotel_id)`：优化根据酒店查询房型的性能
- `reservations(user_id)`：优化根据用户查询预订的性能
- `reservations(hotel_id)`：优化根据酒店查询预订的性能
- `reservations(room_id)`：优化根据房型查询预订的性能
- `reservations(status)`：优化根据状态查询预订的性能
- `guests(user_id)`：优化根据用户查询客人的性能
- `price_calendar(hotel_id, date)`：优化根据酒店和日期范围查询价格的性能

## 4. 结论
